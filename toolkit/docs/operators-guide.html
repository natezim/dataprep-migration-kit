The core insight

A Dataprep flow is a “recipe” of steps written in Wrangle — a finite, declarative DSL. Because the vocabulary is finite, migration is transpilation (apply a reviewed mapping), not a from-scratch rewrite. That is precisely why the output is consistent, predictable, and trustworthy.

In plain EnglishThe old tool only understands a fixed, limited set of commands. So we built a dictionary that says “this command becomes this SQL.” Migrating a flow is just looking each command up in that dictionary and writing down the answer — not inventing new code from scratch. That's why two people get the same result, and why you can trust it.
from-scratch rewrite → Wrangle step ↦ reviewed mapping ↦ SQL CTE

How It Works — The Per-Flow Pipeline

Every flow travels the same seven stages. Each stage produces an artifact the next stage consumes, so the path is repeatable and auditable end-to-end.

1

Extract

Export the flow package (ZIP) via Dataprep API or UI.

2

Inventory

Parse package → backlog, DAG, complexity, target.

3

Translate

Transpile each Wrangle step → one commented CTE.

4

Compile / Dry-run

Catch structure, syntax & cost before running.

5

Parity audit

Run into staging & diff vs the legacy table.

6

Document

Write the explanation + a runnable validation query.

7

Sign-off

A human attests they independently validated it.

teal stage = read-only blue stage = produces team-owned artifacts
1 · Extract

Export a flow package — a ZIP of flow.json + recipes/ + inputs/ + outputs/ — via the Dataprep API (GET /v4/flows/{id}/package) or the UI “Export Flow” button. Both produce the identical ZIP.

In plain EnglishWe download a complete copy of the existing job — every rule it follows and every file it reads and writes — as a single zip. Like photographing a recipe so we can rebuild the dish in a new kitchen. We only copy; nothing in the old tool is changed.
2 · Inventory

Parse the package into a backlog, a dependency graph, a complexity estimate, and a recommended target for the migration.

In plain EnglishWe open the zip and take stock: what does this job actually do, what else does it depend on, how hard will it be, and where should it live in BigQuery? It's the “survey the work before quoting it” step — a to-do list and a map.
3 · Translate

Transpile each Wrangle step into one commented SQL block — a CTE — with the original step quoted inline above it.

In plain EnglishWe rewrite each step of the old recipe as the equivalent step in SQL, one at a time, keeping the original instruction written right above its replacement. Every old step has a known SQL equivalent — nothing is guessed.
4 · Compile / Dry-run

Catch structural, syntactic, and cost problems before a single byte is processed for real.

In plain EnglishBefore running anything for real, we ask BigQuery to check the SQL the way spell-check reads a document — does it make sense, will it run, and roughly what will it cost? Mistakes get caught here, for free, instead of after a slow, expensive run.
5 · Parity audit

Run into a disposable staging dataset and diff the result against the legacy table using the 4-tier engine (§08).

In plain EnglishWe run the new SQL in a throwaway sandbox and compare its output — row by row — against what the old job produces. This is the proof step: if the numbers don't match exactly, it isn't done. The sandbox cleans itself up afterward.
6 · Document

Write the plain-English explanation plus a user-runnable validation query so the team can check the work itself.

In plain EnglishWe write a short, jargon-free explanation of what the flow does and hand you a ready-made query you can run yourself to confirm the new results match the old ones — so you can verify and own it without us.
7 · Sign-off

A human attests they independently validated the flow before it is marked Productionized. Nothing reaches production on the tool's say-so alone.

In plain EnglishA person on your team actually runs the check, reads the SQL, and puts their name to it. Only then is it “live.” The tool never promotes its own work.

● Not started
Plain EnglishThe job has been found and logged, but no one has started migrating it. It's waiting in the queue.
● In process
Plain EnglishSomeone is actively rebuilding this job in SQL right now — translating, compiling, and checking it.
● Validating
Plain EnglishThe new SQL is built and is being compared against the old output to prove the numbers match, while a person reviews the code.
● Parallel runs
Plain EnglishFor a while, the old job and the new SQL run side-by-side on the same schedule, so you can watch them agree on real, live data before retiring the old one.
● Productionized
Plain EnglishA human has signed off. The new SQL is now the official version, and the old Dataprep flow can be switched off.

Transitions are gated. Status lives in status/migration_status.csv (with an Excel view) and is updated as flows progress. Gemini asks you to confirm before advancing any flow, and “Productionized” additionally requires the sign-off attestation (§04 · /dp:signoff).

Agents

The commands orchestrate four specialized agents. Several are strictly read-only on the systems they touch.

@flow-inventoryread-only

Turns a flow package into a backlog, a dependency DAG, a complexity estimate, a recommended target, and a status entry.

Plain EnglishThe surveyor. It reads the old job and writes up what it does, what it needs, and how big a task it'll be — without changing anything.
@recipe-translator

Recipe → ONE commented, self-contained BigQuery SQL file (one CTE per Wrangle step, original step quoted). Also writes validation.sql + EXPLANATION.md. No hard-coded values; applies the three corruption fixes.

Plain EnglishThe translator. It turns the old recipe into clean, heavily-commented SQL, plus a plain-English write-up and a query you can run to check it yourself.
@parity-auditorread-only on legacy

Freezes the input, runs the migrated code into staging, and performs the 4-tier diff against the legacy result.

Plain EnglishThe proofreader. It runs the new code on the exact same frozen data the old job used and checks — old vs new — that every row matches.
@governance

Runs a readiness checklist — no hardcoding, comments present, single self-contained file, parity green, validation.sql + EXPLANATION.md present, naming — producing a governance report that gates sign-off.

Plain EnglishThe inspector. It runs a quality checklist and won't let a job reach sign-off until every box is ticked. It checks; a human still approves.

07

Output Standards

Why the generated SQL is trustworthy and maintainable — these are the rules every flow's output obeys.

SQL-first: one self-contained .sql per flow

A single console-runnable .sql per flow using a “Create-Execute-Clean” pattern:

A

Mount

GCS CSVs as external tables (EXT_*).

B

Transform

A CTE graph, one CTE per Wrangle step.

C

Create

The staging table (STG_*).

D

Clean

Drop the externals so the schema stays pristine.

The Dataform .sqlx wrapper is optional — useful for orchestration, never required to run the migration.

NO hard-coded values

Load/run dates and params become DECLARE vars, CURRENT_DATE(), or query parameters — this is required for automation. A value baked into a string can't be scheduled.

-- ✗ NOT this
WHERE load_date = '2026-06-25'

-- ✓ This — parameterized, automatable
DECLARE run_date DATE DEFAULT CURRENT_DATE();
WHERE load_date = run_date
Heavy commenting

A header block on every file — what / why / when / source / owner / parity result — plus an inline comment on every CTE.

-- ============================================================
-- WHAT:   Daily revenue rollup by region
-- WHY:    Replaces Dataprep flow "rev_daily" (legacy)
-- WHEN:   Scheduled daily 06:00 UTC
-- SOURCE: gs://.../revenue/*.csv  →  STG_revenue_daily
-- OWNER:  analytics-data@
-- PARITY: EXACT MATCH — 4-tier, 2026-06-24
-- ============================================================
Independent-validation artifacts

Every flow also ships:

validation.sql
A query you run to compare new vs legacy yourself.
EXPLANATION.md
A plain-English walkthrough of the migration.

So the team validates independently and owns the result — not the tool.

08

The Verification Engine

This is the trust core. Parity isn't asserted — it's proven against a frozen input with an escalating, exact-match diff.

Freeze the input

Snapshot the source with BigQuery time-travel so the migrated code and the legacy Dataprep run see identical input. This kills false failures from data drift — any difference that remains is a real difference.

In plain EnglishLive data changes constantly, so comparing today's new run against yesterday's old run would show fake differences. Instead we pin both runs to the exact same frozen snapshot of the source — like pausing a video so two people describe the same frame. Any difference left over is therefore real.

4-tier diff — escalating

In plain EnglishWe compare the old and new results in four passes, each stricter than the last: start cheap and broad, and only dig down to the slow, cell-by-cell comparison if something looks off. We stop the moment we've either proven they match or pinpointed exactly where they don't.
1

Schema

Column names, types, and order line up.

cheapest →
2

Row count

Same number of rows produced.

↓ escalate
3

Hash each row and compare the two multisets — catches any content difference regardless of order.

↓ escalate
4

Cell-level

Pinpoint the exact key + column that differ — so a failure is immediately actionable.

→ most precise

Normalize known-legitimate differences first — column order, float rounding, row order, timezone, surrogate keys — then the bar is exact match. Nothing legitimate is allowed to mask a real defect, and nothing cosmetic is allowed to fail the audit.

The three silent-corruption fixes — always applied

a · Temporal

Date-shift

Dataprep dates are timezone-naive strings. Keep them tz-naive in SQL to avoid a silent date-shift across timezones.

Plain EnglishThe old tool stores dates as plain text with no timezone. If SQL “helpfully” assumes one, a date can jump a day. We keep dates timezone-free so they stay put.
b · Decimal

Precision cap

Cap precision at 38. Alteryx allows 50; BigQuery caps at 38 — overflowing it corrupts values silently.

Plain EnglishThe old tool allowed more digits than BigQuery does. Numbers that don't fit get silently mangled, so we trim them to a size BigQuery can hold exactly.
c · Null propagation

Coalesce first

Coalesce before string concatenation. Wrangle treats null == empty string; SQL-92 does not, so a null poisons the whole concat.

Plain EnglishIn the old tool a blank acts like empty text. In SQL, a blank “poisons” anything you glue it to — turning the whole result blank. We convert blanks to empty text before joining text together.
09

Safety & Access Model

The toolkit is read-only by construction. Here is exactly what it can and cannot touch.

Dataprep is READ-ONLY

The toolkit only GETs — list + export. It never creates, edits, deletes, or runs anything in Dataprep. Migration is strictly one-directional.

Production (BigQuery) is read-only

SELECT-only on source / legacy. Never DDL, never DML against production.

All writes go to ONE disposable staging dataset

dataprep_migration_staging with a default table expiration, so it self-cleans. A write-guard refuses any non-staging write.

gcloud not required

Use the Python BigQuery client with browser OAuth, or the BigQuery console / Dataform UI. No CLI install needed.

API-optional

Business units without API access use the UI export — everything downstream is identical.

12

Governance & Ownership

The closing principle — and the reason this approach holds up.

The team owns the migration; Gemini assists.

Nothing is “Productionized” on Gemini's say-so. A human runs the validation query, reviews the SQL, and signs off. @governance checks quality; /dp:signoff records the human attestation.

quality gate

@governance runs the readiness checklist and produces a report — but it gates, it does not approve. A failing check blocks sign-off.

human attestation

/dp:signoff is the human stating, on the record, that they reviewed and validated the flow themselves. No rubber-stamping.

The toolkit keeps improving in place via a documented hardening loop: every new finding is captured back into the reference files, so the next migration starts smarter than the last.
